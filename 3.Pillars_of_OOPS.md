1. Abstraction

# 1. Abstraction (Implementation Hiding)

### Definition

Abstraction hides the **internal implementation** and exposes **only the essential functionality** to the user.

> **Implementation hiding** = User knows **what** to do, not **how** it is done.

### Achieved Using

* Interface (100% abstraction before Java 8; now supports default/static methods)
* Abstract Class (0–100% abstraction)

### Advantages

* Hides implementation details
* Improves security
* Reduces complexity
* Easier maintenance

### Real-life Example

**Car Brake**

```text
Brake Pedal (Visible)
        ↓
Press Brake
        ↓
Car Slows Down
```

The user only presses the brake pedal.

How the hydraulic system, brake pads, and ABS work is hidden.

> **Memory Trick:** "Aam khao, guthli mat gino." 🍋

---

## Interface Example

```java
interface Car {

    void applyBrake();

    void pressHorn();
}

class CarImplementation implements Car {

    @Override
    public void applyBrake() {
        // Actual implementation
    }

    @Override
    public void pressHorn() {
        // Actual implementation
    }
}
```

### Interview One-liner

> Abstraction hides **implementation** and exposes **functionality**.

# 2. Encapsulation (Data Hiding)

### Definition

Encapsulation is the process of **binding data (variables)** and **methods (behavior)** together into a single unit (class).

It also **hides the data** from direct outside access.

### Achieved By

1. Making variables **private**
2. Providing **public getters and setters**

```java
class Employee {

    private int salary;

    public int getSalary() {
        return salary;
    }

    public void setSalary(int salary) {
        if (salary > 0)
            this.salary = salary;
    }
}
```

---

## Why Private?

Without encapsulation

```java
employee.salary = -10000;
```

Anyone can modify it.

With encapsulation

```java
setSalary(-10000);
```

Setter validates the value before updating.

---

## Benefits

* Data Hiding
* Data Validation
* Better Security
* Read-only classes (only getter)
* Write-only classes (only setter)
* Easy Maintenance
* Loose Coupling

---

## Access Modifiers

| Modifier  | Same Class | Same Package | Subclass | Other Package |
| --------- | ---------- | ------------ | -------- | ------------- |
| private   | ✅          | ❌            | ❌        | ❌             |
| default   | ✅          | ✅            | ❌        | ❌             |
| protected | ✅          | ✅            | ✅        | ❌*            |
| public    | ✅          | ✅            | ✅        | ✅             |

* `protected` is accessible in subclasses even if they are in a different package.

---

### Interview One-liner

> Encapsulation hides **data**, whereas Abstraction hides **implementation**.

---

# Difference: Abstraction vs Encapsulation

| Abstraction                               | Encapsulation                                      |
| ----------------------------------------- | -------------------------------------------------- |
| Hides implementation                      | Hides data                                         |
| Achieved using Interface & Abstract Class | Achieved using private variables + getters/setters |
| Focuses on **what**                       | Focuses on **how data is protected**               |

---

# 3. Inheritance

### Definition

Inheritance allows one class (child) to acquire the **properties** and **behaviors** of another class (parent).

### Advantages

* Code Reusability
* Reduces code duplication
* Easier maintenance
* **Enables Runtime Polymorphism**

---

## Types of Inheritance

### Single

```text
A
│
B
```

---

### Multilevel

```text
A
│
B
│
C
```

---

### Hierarchical

```text
    A
   / \
  B   C
```

---

### Multiple (Not Supported with Classes)

```text
A      B
 \    /
   C
```

Reason:

**Diamond Problem (Ambiguity)**

If both A and B have the same method, Java wouldn't know which one to inherit.

---

### Multiple Inheritance Using Interfaces

```java
interface A {}

interface B {}

class C implements A, B {

}
```

---

### Hybrid Inheritance

Possible **only through interfaces**.

```text
Interface A      Interface B
       \         /
        \       /
         Class C
             |
          Class D
```

---

## Summary

| Type         | Supported?     |
| ------------ | -------------- |
| Single       | ✅              |
| Multilevel   | ✅              |
| Hierarchical | ✅              |
| Multiple     | ❌ (Classes)    |
| Hybrid       | Via Interfaces |

---

### Interview One-liner

> Java supports **Single, Multilevel, and Hierarchical** inheritance with classes. Multiple and Hybrid inheritance are achieved using **interfaces**.

# 4. Polymorphism

### Definition

**Poly** = Many

**Morph** = Forms

One method behaves differently in different situations.

---

## Types

### 1. Compile-time (Static) Polymorphism

Achieved using

**Method Overloading**

Compiler decides which method to call.

```java
class Calculator {

    int add(int a, int b) {
        return a + b;
    }

    int add(int a, int b, int c) {
        return a + b + c;
    }
}
```

---

### Invalid Overloading

Changing **only return type** is NOT overloading.

```java
int add(int a, int b) {
    return a + b;
}

String add(int a, int b) { // ❌ Compile-time Error
    return "";
}
```

Reason:

Compiler identifies overloaded methods by **method signature**, not return type.

Method Signature =

* Method Name
* Number of Parameters
* Type of Parameters
* Order of Parameters

---

### 2. Runtime (Dynamic) Polymorphism

Achieved using

**Method Overriding**

Child class provides its own implementation.

```java
class Animal {

    void sound() {
        System.out.println("Animal makes a sound");
    }
}

class Dog extends Animal {

    @Override
    void sound() {
        System.out.println("Dog barks");
    }
}

public class Main {

    public static void main(String[] args) {

        Animal a = new Dog();

        a.sound(); // Dog barks
    }
}
```

If `Dog` does **not** override `sound()`, then the inherited method from `Animal` is executed.

---

## Rules for Method Overriding

* Same method name
* Same parameters
* Same (or covariant) return type
* Child class inherits from parent
* Cannot reduce method visibility (e.g., `public` → `private`)

---

## Overloading vs Overriding

| Method Overloading               | Method Overriding            |
| -------------------------------- | ---------------------------- |
| Compile Time                     | Runtime                      |
| Same class (usually)             | Parent & Child class         |
| Different parameters             | Same parameters              |
| Return type doesn't matter alone | Same/covariant return type   |
| Increases readability            | Enables runtime polymorphism |

---

# OOP Interview Revision (30 Seconds)

### Abstraction

* Hides implementation.
* Achieved using Interface & Abstract Class.

### Encapsulation

* Hides data.
* Achieved using private variables + getters/setters.

### Inheritance

* Acquires properties from parent class.
* Improves code reuse.
* Enables runtime polymorphism.

### Polymorphism

* One method, many forms.
* Overloading → Compile Time
* Overriding → Runtime
